# Desktop Application

Cross-platform desktop application built with [Wails v2](https://wails.io) (Go backend), [Preact](https://preactjs.com), and TypeScript. The frontend toolchain is Bun-only — Bun handles package management, bundling, dev server, and testing. No Vite, no Webpack, no Node.js.

## Prerequisites

- [Go](https://go.dev/dl/) 1.21+
- [Bun](https://bun.sh)
- [just](https://github.com/casey/just)
- [Wails CLI](https://wails.io/docs/gettingstarted/installation)

```sh
go install github.com/wailsapp/wails/v2/cmd/wails@latest
```

Verify your environment:

```sh
just doctor
```

## Quick Start

```sh
just install   # install frontend dependencies
just dev       # start development mode with hot-reload
```

## Project Structure

At a high level:

- Repository root: Go/Wails application entrypoint, backend code, build metadata, and top-level project tooling such as `justfile` and `wails.json`.
- `frontend/`: Bun-managed frontend workspace, including package metadata, Bun config, and the HTML entrypoint.
- `frontend/src/`: Preact application source, including app bootstrapping, views, reusable components, styling, and frontend build/dev scripts.
- `frontend/src/test-support/`: shared frontend test setup and test-only utilities.
- `frontend/src/wailsjs/`: checked-in TypeScript stubs for Wails bindings so frontend development and type-checking work before generation.
- `frontend/wailsjs/`: generated runtime bindings emitted by Wails during build/dev.
- `build/`: application assets and platform-specific packaging metadata.

`frontend/wailsjs/` is auto-generated by Wails and contains the runtime bindings used at build/dev time. The checked-in stubs under `src/wailsjs/` keep TypeScript happy before the first generation and during isolated frontend work.

## Architecture

### Bun-Only Frontend Toolchain

The frontend uses Bun for everything:

| Concern | Tool |
|---|---|
| Package manager | `bun install` |
| Bundler | `Bun.build()` with HTML entrypoint |
| Dev server + HMR | `Bun.serve()` dev proxy in front of an upstream Bun HMR server |
| JSX/TSX | Bun runtime (native, zero-config) |
| CSS | Bun CSS bundler (native) |
| Test runner | `bun test` (`bun:test`) |
| Component testing | Testing Library for Preact + happy-dom |
| Test mocking | `mock()` from `bun:test` |

No Vite, Vitest, Webpack, esbuild, or Node.js required.

### Go Backend (`app.go`)

The `App` struct is bound to the frontend via Wails. Exported methods become callable from TypeScript:

| Method | Signature | Description |
|---|---|---|
| `Greet` | `(name string) string` | Returns a greeting string |
| `GetSystemInfo` | `() map[string]string` | Returns OS, arch, CPU count, Go version |
| `OpenDirectoryDialog` | `(title string) (string, error)` | Opens a native directory picker |
| `OpenFileDialog` | `(title string) (string, error)` | Opens a native file picker |
| `SetTitle` | `(title string)` | Dynamically sets the window title |

Lifecycle hooks `startup` and `shutdown` are registered in `main.go` and receive the Wails runtime context.

### Frontend Bindings

Wails generates JavaScript bindings at `frontend/wailsjs/go/main/App.js` (with `.d.ts` declarations). Import them in components:

```ts
import { Greet, GetSystemInfo } from "../wailsjs/go/main/App";

const result = await Greet("world");
```

### Window Configuration (`main.go`)

- Default size: 1024x768, minimum 800x600
- macOS: transparent titlebar with hidden native title, full-size content, About panel
- Windows: standard window decorations with WebView2
- Linux: standard GTK window

### Routing

State-based routing in `app.tsx` using a `Route` union type and `useState`. No router library needed — the `NavigationShell` component receives the current route and an `onNavigate` callback.

### Theming

CSS custom properties in `frontend/src/assets/style.css` with automatic light/dark switching via `prefers-color-scheme`. The design tokens cover backgrounds, text, borders, and accent colors.

## Testing

### Go Backend

Tests are in `app_test.go`, run with `go test` via the justfile. They cover:

- `NewApp` initialization and zero-value state
- Lifecycle hooks (`startup` stores context, `shutdown` is safe)
- `Greet` with table-driven cases (simple names, empty input, unicode)
- `GetSystemInfo` key presence and value correctness

Methods that require the Wails runtime context (dialogs, SetTitle) are not unit-testable without the full runtime and are instead tested via the frontend through prop injection.

### Frontend

Tests use `bun:test` as the runner, [happy-dom](https://github.com/capricorn86/happy-dom) as the DOM environment, and Testing Library for rendering, queries, and async waiting. Wails bindings are injected as props rather than module-mocked, making tests fast and deterministic.

**NavigationShell** (11 tests):
- Renders titlebar and navigation items
- Active route highlighting
- Sidebar collapse/expand toggle
- Collapsed label abbreviation (first character)
- Calls `onNavigate` callback
- Wails drag region attributes
- Renders children in content area

**HomeView** (8 tests):
- Renders heading, subtitle, input, and button
- Calls `greet` prop on button click and enter key
- Displays the greeting result
- Guards against empty/whitespace input

**SystemView** (7 tests):
- Loading state before data arrives
- Renders system info cards from `getSystemInfo` response
- Handles `getSystemInfo` failure gracefully
- Calls `openDirectoryDialog` and displays selected path
- Handles dialog cancellation and failure

## Development Workflow

`just dev` starts Wails development mode, which in turn coordinates the frontend and backend dev loops:

1. **Public Bun dev proxy** on `http://localhost:34115` — this is the URL Wails loads during development.
2. **Upstream Bun HMR server** on `http://localhost:34116` — this serves Bun's native HMR/client assets.
3. **Wails dev watcher** — recompiles the Go backend on file changes and restarts the application window.

The proxy layer exists so Wails can use Bun HMR cleanly while preserving dev sourcemaps and fixing the custom `wails://` origin edge cases. The `wails.json` config points Wails at the public proxy via `frontend:dev:watcher` (`bun run dev`) and `frontend:dev:serverUrl` (`http://localhost:34115`).

## Adding a New Backend Method

1. Add an exported method to the `App` struct in `app.go`:

```go
func (a *App) MyMethod(arg string) (string, error) {
    return "result", nil
}
```

2. Run `just generate` (or `just dev` which does this automatically).

3. Import and call from the frontend:

```ts
import { MyMethod } from "../wailsjs/go/main/App";
const result = await MyMethod("input");
```

## Adding a New View

1. Create `frontend/src/views/my-view.tsx`:

```tsx
export function MyView() {
  return <div><h1>My View</h1></div>;
}
```

2. Add it to the route switch in `frontend/src/app.tsx`:

```tsx
import { MyView } from "./views/my-view";

// Add to the Route type
export type Route = "/" | "/system" | "/my-view";

// Add to the switch
case "/my-view":
  view = <MyView />;
  break;
```

3. Add a nav entry in `frontend/src/components/navigation-shell.tsx`:

```tsx
const navItems: { path: Route; label: string }[] = [
  // ...existing items
  { path: "/my-view", label: "My View" },
];
```
